## Component Inventory

New components required for this phase:

### Sub-nav row (inline in JobList.tsx or App.tsx)
- Two tab buttons: "Queue (N)" and "History (N)"
- Tab switcher style: underline indicator (2px solid `var(--blue)`) beneath active tab; no pill/background fill
- Active tab: text color `var(--txt)`, weight 600
- Inactive tab: text color `var(--txt-3)`, weight 400
- Hover inactive: text color `var(--txt-2)`
- Tab buttons use `background: transparent`, `border: none`, `cursor: pointer`
- "Clear completed" and "Clear failed" buttons render at the right end of the sub-nav row, visible only when History tab is active
- Disabled state (count = 0): `opacity: 0.4`, `cursor: not-allowed` — do NOT hide (zero-count context is useful feedback)

### BulkActions.tsx (sub-nav right side)
- "Clear completed" button: ghost style matching existing utility buttons (raised bg, border, `var(--txt-2)` text, 38px height)
- "Clear failed" button: same ghost style
- Both open a Radix `AlertDialog.Root` confirmation before calling the bulk API
- Confirmation copy: see Copywriting Contract below

### HistoryList.tsx
- Same panel container as `JobList.tsx`: `background: var(--panel)`, `border: 1px solid var(--border)`, `borderRadius: 8`, `overflow: hidden`
- Column headers: File | Status | Avg VMAF | Avg CRF | Duration | Actions — grid `1fr 120px 110px 90px 110px 140px`
- Rows use the same `JobRow` component with a new `isHistory` prop
- History rows show: filename (mono 13px), StatusBadge (DONE or FAILED), avgVmaf (mono 12px), avgCrf (mono 12px), duration formatted as Xh Xm Xs (mono 12px), [Retry] [Delete] buttons
- Left border uses `STATUS_BORDER[job.status]` (existing map — DONE = #22c55e, FAILED = #ef4444)

### Delete button (in JobRow, history variant)
- Height 28px, padding `0 12px`, fontSize 12, fontWeight 400, borderRadius 4
- Color: #fca5a5 (matches CancelDialog trigger text color exactly)
- Background: #220f0f
- Border: `1px solid #7f1d1d60`
- Opens `DeleteJobDialog` on click

### DeleteJobDialog.tsx
- Reuses the `AlertDialog.Root` pattern from `CancelDialog.tsx` exactly
- Dialog dimensions: 340px width, padding 24px
- Icon: 40x40px icon box, background #220f0f, border `1px solid #7f1d1d`, contains trash SVG (16x16) in #ef4444
- Title: 15px, weight 600, `var(--txt)`
- Description shows the filename in monospace so the user knows exactly what is being removed
- Buttons: [Keep] ghost + [Delete job] destructive (background #b91c1c, border #991b1b, white text, weight 600)

### SettingsModal — Retention section (appended to existing modal)
- New `<h3>` section heading "Retention" using the existing uppercase 10px/600 heading style
- Single numeric input: label "Auto-remove completed jobs after", input (days, integer, min 0), "days" suffix label
- Helper text below input: "Set to 0 to disable. Default: 7 days." — 12px, `var(--txt-3)`
- Follows the same `numField` pattern already in SettingsModal

## Interaction Contract

### Tab switching
- Tab state: React `useState<"queue" | "history">` in the parent that owns both `JobList` and `HistoryList`. No router, no URL change.
- Switching to History tab triggers a one-time fetch if history jobs are not already in the Zustand store (they are — same `jobs` array, filtered by status)
- No re-fetch loop on History tab — reads the existing Zustand `jobs` array filtered to `status === "DONE" || status === "FAILED"`
- Tab counts are derived from Zustand store array length — live, no separate count fetch

### Delete single job
1. User clicks [Delete] on a history row
2. `DeleteJobDialog` opens (AlertDialog)
3. User clicks [Delete job] to confirm
4. Call `DELETE /api/jobs/{id}`
5. On 200: remove job from Zustand store via `removeJob(id)` action
6. Row animates out (same `AnimatePresence` + `motion.div` pattern used by JobCard expand/collapse — duration 0.18s, easeInOut)
7. On error: dialog closes, show no toast (silent failure acceptable for v1.1 — no toast system exists)

### Bulk clear
1. User clicks [Clear completed] or [Clear failed]
2. Bulk confirmation AlertDialog opens showing exact count
3. User clicks [Clear all] to confirm
4. Call `DELETE /api/jobs/bulk` with body `{ status: "DONE" }` or `{ status: "FAILED" }`
5. On 200: remove all matching jobs from Zustand store
6. History list re-renders; if now empty, shows History empty state

### Auto-cleanup
- Background task only — no UI interaction required beyond the settings field
- When `auto_cleanup_days > 0`, completed jobs older than that threshold disappear from the Zustand store on next poll
